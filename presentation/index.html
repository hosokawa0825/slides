各FWのメリット・デメリット

Backbone.js

メリット

  • コミュニティが大きい。StackOverflowにノウハウが蓄積されている。FWの問題点はコミュニティによってすぐ解決される。継続的に開発されている。
  • 実績は最も豊富（SoundCloud、LinkedIn、DocumentCloudで使われている）
  • ドキュメント豊富。
  • シンプル、軽量。コアコンポーネントはコレクション、モデル、ビュー、ルーターの４つだけ。
  • シンプルなのでアーキテクチャを変更しやすい。
  • RESTfulAPIを使ってサーバ上のデータを操作できる。
  • モデル、ビュー、イベント、テンプレート、ルーティング機能がある。
  • モデルとビューをバインド出来る。
  • MVVMパターン
  • Marionette、VertebraeなどのBackbone.jsをベースとしたFWが作られている。
  • おまじないを沢山書く必要があるが、そのかわり黒魔術は少なめ。好きなようにアーキテクチャを組める。
  • Underscore,jQuery,Zeptoと統合されている。
  • シンプルで、チュートリアルが豊富なため勉強しやすい。とりあえずMVCフレームワークを勉強してみたいならコレ。

Backbone.js

デメリット

  • extendメソッドがいまいち。これは親オブジェクトのプロパティを子オブジェクトにコピーする。プロトタイプ継承を使わない。
  • 変化が激しいため、一ヶ月前の記事やチュートリアルに書かれている内容がもう使えなかったりする。
  • ビューとモデルのバインディングを手動で管理する必要がある。モデルの一部が変更されたらビュー全体が更新されてしまう。
  • RESTfulAPIを使って複数件のデータを一括登録・更新することができない。（これはRESTの問題ではあるが）

Ember.js

メリット

  • 実績豊富（Square Inc.など）
  • ドキュメント豊富。
  • テンプレートエンジンのHandlebarsと統合されている。
  • ネイティブアプリぐらいリッチなWebアプリが作れる。
  • 目的特化型
  • CoCによりモデルとビューのバインドが楽に出来る。コードが少なくて済む。
  • MVCパターン
  • RESTfulAPIを使ってサーバ上のデータを操作できる。
  • 開発者のYehuda Katz(Rails Core Teamの人)は実績があるので信頼できる。

デメリット

  • バージョンが1.0未満なので今後仕様が変わる可能性がある。特にrouterとEmber data
  • Backbone.jsなどに比べてドキュメントが不足している。FWのコードを調べて使い方を把握する必要がある。
  • チュートリアルなどの学習リソースも少ない。入門者はつらい。
  • 黒魔術使いまくりなのでちょっと怖い。分析が難しい不具合が起こるかも。。
  • エラーメッセージがわかりにくい。
  • コミュニティが小さい。
  • 永続化機能が弱い。localStorageを使ったり、RESTfulAPI経由でデータをサーバと同期したりする機能が無い。

JavaScriptMVC

メリット

  • 実績豊富（IBMで使われている）
  • 開発に必要なツールが一通り入ってる。

デメリット

  • FWの開発スピードが遅い

CanJS

メリット

  • 軽量
  • 自動更新テンプレート
  • jQuery,Dojoのようなメジャーなライブラリと統合されている
  • パフォーマンスがいい
  • モデル、ビュー、コントローラ、ルーティング機能がある。
  • 大きなアプリが作られた実績はないが、今後作られる可能性はある
  • 開発チームが経験豊富

AngularJS

メリット

  • 宣言したらViewの振る舞いも定義される。これはカスタムHTMLタグとViewクラスで実現されている。
  • 簡単にテストできる。
  • モデル、ビュー、コントローラ、ルーティング、テンプレート機能がある。
  • Web Componentsのような先進的なWebプラットフォームの機能を取り入れている
  • アプリケーションテンプレート生成ツールがある(angular-seed)
  • 双方向データバインディングが強力。イベントを気にせずモデルのデータを正しくすることに集中できる。
  • FWのファイルサイズが小さい。テンプレートエンジンは文字列をDOMに変換するのではなく、DOMを直接テンプレートとして使用するのでパフォーマンスがいい。　TODO　要確認
  • モダンブラウザのみサポートする場合はjQueryがなくても動く。
  • わかりやすい、使いやすい。
  • ドキュメントが豊富。
  • スコープがDOMと一致している。　TODO　要確認
  • Chrome拡張のBatarangがChrome Developer’s Toolsに統合されていてデバッグ時にデータを確認しやすい。　TODO　要確認
  • 定型的なコードがかなり減る。
  • クラスの再利用がしやすい。　TODO　要確認
  • カスタムタグによって複雑な機能が簡単に実現できる。
  • DIによって簡単にテストできる。　TODO　要確認
  • Backbone.jsに比べて使えるようになるまでが早い。

デメリット

  • UIの状態を明示的に（自分が書いたコードによって）変更できない。特に、ng-show属性のエレメントをfede inとかのアニメーションを使って表示させることが出来ない　TODO　要確認

Dojo

メリット

  • 様々なwidgetを作れる（画像、グラフ、グリッド）
  • JSファイルの遅延ロード機能がある。
  • CDNとの統合。
  • 国際化機能が強い。
  • クラスシステムサポート。
  • このFWを土台として新しいFWを作れる。
  • シンタックスがシンプル。
  • 他のJSファイルを読み込む起点になるローダーファイルの大きさが6KB程度と小さい。
  • ビルド機能がある。
  • AMDにより依存関係を解決する機能がある。
  • DOMの操作からSVG,VML,canvasまで幅広く多くの機能がある。
  • ウィジェットシステムのDijitは使いやすさと拡張性が素晴らしい。
  • UIコンポーネントがいい。タブ、グリッド、フォームなど。クロスブラウザ対応されている。
  • モック（mocks.Integrates）を使って簡単にテストできる。

デメリット

  • dojo/_base/declareの機能はstrictモードに100%準拠しているわけではない。後方互換性のためにパフォーマンスが多少犠牲になっている。これらの問題のほとんどはDojo2.0では解決される予定ではある。
  • UIコンポーネントのデフォルトのデザインが時代遅れ。
  • HTML5対応でない部分がある。
  • ドキュメントの量はまずまず。
  • テンプレートエンジンは弱い。自動でデータをバインドしてくれない　TODO　要確認

YUI

メリット

  • YUI3は機能がモジュール化されており、必要な機能だけ取り出して他のライブラリと一緒に使える。
  • 疎結合化の機能がBackbone.jsより優れている。例えば、イベント関係の機能はイベントモジュールをmixinすることでどのオブジェクトにも追加できるようになっている。
  • Ember.jsと連携する機能がある。
  • YUIエクステンションが使える。
  • モデル、ビュー、ルーティング機能がある。
  • 異なるURLを持つ複数のViewを持つアプリを簡単に作れる。　TODO　要確認
  • ビュートランザクションのサポート　TODO　要確認
  • widget、ビューコンポーネント、アーキテクチャ作成をサポートする仕組みを含む全部入りFW

デメリット

  • テンプレート作成ツール（yuiproject）はあるが、あまりおすすめできない
  • ファイルサイズは大きい
  • AMD機能がない。モジュールローダーを使って必要なJSファイルだけダウンロードして使うといったことができない

Spine

メリット

  • シンプルで依存するライブラリがない
  • 非同期API
  • 目的特化型だが、柔軟性もある。
  • シンプルなモデル、ビュー、コントローラ、イベント、ルーティング
  • CoffeeScriptと統合されている
  • ドキュメントが豊富

KnockoutJS

メリット

  • 複雑で、動的なUIを作れる
  • モデルとビューの双方向バインディングによってビューが自動的に更新される
  • モデルデータのdependency trackingがサポートされている（？？？）
  • 好きなFWと一緒に使える
  • 既存のアプリに組み込むことも簡単
  • テンプレートエンジンを持つ
  • 拡張しやすい
  • Collection Viewを使うとViewのソートが簡単にできる。
  • カスタムイベントが利用できる
  • データバインディングが使いやすい
  • テンプレートの代わりにコントロールフローバインディングを使っているところがいい　TODO　要確認

デメリット

  • アプリの規模が大きい場合、APIが使いづらくなりそう。　TODO　要確認
  • アプリを構造化するのが難しい。ビューモデルはあるが、正しく構造化されたアプリを作るのが難しい。

jQuery

メリット

  • クロスブラウザ対応
  • イベントが使える
  • 外部サービスと連携するのに使える
  • 拡張可能
  • プラグインが豊富

デメリット

  • MVCパターンによるコードの構造化はできない
  • 複雑なWebアプリを作るのには適さない

Maria

メリット

  • ピュアMVCフレームワーク。シンプル。

デメリット

  • テストコード、ドキュメントが少ない
  • チュートリアルが少ない

Cujo

メリット

  • 自分好みのMVCフレームワークが作れる。アプリの要件にフィットしたフレームワークを作るためのローレベルなツールが用意されている。
  • プラグインを作るための機能が豊富。プラグインは他のMVCフレームワークを統合したり、拡張したりするのに使える。

デメリット

  • コミュニティが小さい。大規模なアプリを作るノウハウが足りない
  • ドキュメント、チュートリアル、サンプルが少ない。が、ドキュメントはもうすぐ増えそう。

Extjs

メリット

  • Sencha Architectというツールによって、UIをGUI環境で作れる。
  • モバイル用FWのSencha Touchがあり、Extjsと似ているためモバイル用アプリを作りやすい。
  • 複雑なUIを比較的簡単に作れる。
  • 業務システムを作るにはベストなフレームワークと思われる。
  • 使えるようになるまでは比較的早い。
  • MVCパターンを使うよう半ば強制されるのでアプリの構造がきれいになりやすい。パッケージ構造とフォルダ・ファイル構成が一致する。
  • UIコンポーネントが非常に豊富。生のHTMLを書くことはほとんどない。

デメリット

  • HTMLを基本的にすべてフレームワークが生成するため、どんなHTMLが生成されるかが把握しづらい。SEOのためにHTMLを最適化するといったことがしづらい。CSSフレームワークが使いづらい。
  • ライセンスの条件は厳しく、わかりづらい。
  • パフォーマンスが悪い。
  • ほぼすべてのHTMLをJavaScriptで生成するため、そのままではクローラブルでない（Googleなどの検索エンジンから検索できない）

Batman

メリット

  • Viewのバインディングが非常に簡単
  • Railsと相性がいい
  • CoCを使っていてコードが少なくて済む

デメリット

  • ドキュメントが少ない
  • 今後開発が順調に進むかが疑問

まとめ

・色んなFWについて勉強しよう

沢山のFWについて勉強するのは大変だが、それによって色んなテクニックを知ることが出来る。

沢山のFWに詳しくなると、作るアプリの要件により近いFWを選べるようになる。それによってより実装が楽になる。

・FWをカスタマイズしよう

フレームワークを使っているとそれにしばられがちで、無理矢理フレームワークの部品を使って課題を解決しようとしてしまう。フレームワークを拡張してもっと適切な実装ができないかを考えるべき。

・FWの選定に時間をかけよう

安定で、メンテナンス性の高いアプリを作るために、時間をかけてFWを選ぶことが大事。
